# UX/UI Overhaul — Specification

> Спецификация переработки интерфейса `constructor-front` (Next.js 16 / React 19 / Tailwind v4 / shadcn-ui).
> Этот документ НЕ заменяет [SPEC.md](./SPEC.md) (W9 Risk Enforcement) — он описывает отдельный фронтенд-трек: навигация, флоу запуска стратегий и общая «гигиена» интерфейса.

**Статус:** Draft — ожидает подтверждения перед реализацией.
**Дата:** 2026-06-22.

---

## 1. Objective (цель и пользователи)

### Проблема
Сейчас навигация перегружена: в шапке висят все разделы сразу (`Strategies`, `Auto-trade`, `Trade`, `Monitor`, `Forecasts`, `Settings`, `Admin`, `Logout`). Флоу «собрал стратегию → бэктест → sandbox → live» размазан по нескольким экранам без явного пути и статуса. На страницах много технических подписей и дублирующего текста, из-за чего интерфейс выглядит «инженерным», а не продуктовым.

### Цель
Сделать интерфейс **понятным и спокойным**: чистая шапка, один очевидный путь запуска стратегии с прозрачными стадиями sandbox → live, и единый визуальный язык на всех страницах. Глубина — **рефреш визуала + реорганизация навигации/флоу** (без полного редизайна с нуля; текущая тёмная тема и shadcn-ui сохраняются).

### Пользователи
- **Трейдер (основной):** собирает/тестирует стратегии, запускает авто-трейд, следит за KPI и рисками.
- **Админ:** доступ к рантайм-снапшоту и конфигу AI-бэктестов (раздел спрятан в меню, гейтится `useAuthStore().hasAdminAccess`).

### Принципы дизайна
1. **Прогрессивное раскрытие** — продвинутые риск-параметры и технические поля скрыты по умолчанию (collapsible / «Advanced»).
2. **Один путь к действию** — на каждом экране одно очевидное главное действие (primary CTA), остальное вторично.
3. **Статус вместо текста** — состояние стратегии передаём бэйджами/индикаторами, а не абзацами.
4. **Меньше слов** — убираем подписи, очевидные из контекста; оставляем только то, что несёт смысл или предупреждает о риске.

## 2. Scope (что входит и не входит)

### Входит
- **Header:** новая шапка — лого + 3 основных раздела + меню профиля/навигации справа.
- **Strategy Launch Flow:** пошаговый визард `выбор стратегии → бэктест → подключение биржи → sandbox → KPI-gate → промоушен в live`.
- **Sandbox/Live lifecycle:** явные стадии и статус-трекер стратегии (sandbox / eligible / live / paused).
- **Гигиена страниц:** ревизия всех экранов на лишний текст, дублирование, пустые состояния (empty states), единые отступы/типографика/карточки.
- **Визуальный рефреш:** токены отступов, иерархия заголовков, состояния (loading/empty/error), консистентные карточки и бэйджи.

### Не входит
- Полный редизайн с новой дизайн-системой с нуля.
- Изменения бэкенд-API (`constructor`). Работаем строго в рамках текущего OpenAPI-контракта.
- Новая бизнес-логика рисков (это трек SPEC.md / бэкенд).
- Замена библиотек (остаёмся на shadcn-ui + Tailwind v4 + Zustand + TradingView charts).

## 3. Header (новая навигация)

### Целевая структура
```
┌─────────────────────────────────────────────────────────────────────┐
│  [Logo]   Strategies   Auto-trade   Monitor          [● status] [▾]   │
└─────────────────────────────────────────────────────────────────────┘
```

- **Видимы всегда (desktop):** Лого + 3 раздела — **Strategies**, **Auto-trade**, **Monitor**.
- **В выпадающем меню справа (`[▾]`, dropdown-menu):**
  - Trade (ручная торговля)
  - Forecasts
  - Settings → Security / Connect exchange / Notifications
  - Admin (только если `hasAdminAccess`)
  - Разделитель
  - Logout
- **Индикатор статуса (`[● status]`):** компактный health/sync-индикатор (зелёный/жёлтый/красный) — переносит сюда то, что сейчас в `app-status-footer`. Клик → поповер с деталями sync/freshness.
- **Mobile (< md):** лого + один гамбургер (`Sheet`), внутри — все разделы + меню профиля списком.

### Технические решения (на основе context7)
- Активный раздел подсвечиваем через `useSelectedLayoutSegment()` из `next/navigation` в client-компоненте `NavLink` (а не ручным сравнением `usePathname`).
- Навигация — `next/link` `<Link>` с дефолтным prefetch для трёх основных разделов; для второстепенных ссылок в меню допустимо `prefetch={false}`.
- Выпадающее меню — shadcn `DropdownMenu` (`DropdownMenuGroup` / `DropdownMenuSeparator` / `DropdownMenuLabel`).
- Mobile-меню — shadcn `Sheet`.

### Затрагиваемые файлы
- `components/layout/app-header.tsx` — переписать структуру.
- `components/layout/app-status-footer.tsx` — статус переносится в хедер (footer упрощается или удаляется).
- Новый: `components/layout/nav-link.tsx` (client, активная подсветка).
- Новый: `components/layout/profile-menu.tsx` (dropdown), `components/layout/mobile-nav.tsx` (sheet).
- `app/(app)/layout.tsx` — подключить новую шапку.

### Acceptance criteria
- [ ] В desktop-шапке видны ровно: лого + 3 раздела + статус-индикатор + меню-триггер. Больше прямых ссылок нет.
- [ ] Текущий раздел визуально подсвечен и определяется через `useSelectedLayoutSegment`.
- [ ] Все второстепенные пункты (Trade, Forecasts, Settings, Admin, Logout) доступны из меню за ≤ 2 клика.
- [ ] Admin-пункт скрыт, если `hasAdminAccess !== true`.
- [ ] На мобильном вся навигация доступна через один `Sheet`-гамбургер.
- [ ] Клавиатурная доступность: меню открывается/закрывается и навигируется с клавиатуры (radix focus management).

## 4. Strategy Launch Flow (sandbox → live визард)

### Модель
Пошаговый визард для **первого запуска новой стратегии**. Управление уже запущенными — на дашборде Auto-trade со статус-бэйджами (визард для них не обязателен).

### Шаги визарда
```
[1] Стратегия → [2] Бэктест → [3] Биржа → [4] Sandbox → [5] KPI-gate → [6] Live
```

| Шаг | Содержание | Главное действие |
|-----|-----------|------------------|
| 1. Стратегия | Выбор типа (VWAP / ATR-OB / Knife / Grid / Intraday / Portfolio) и параметров. | «Дальше» |
| 2. Бэктест | Прогон + результат (equity-кривая, winrate, max DD, Sharpe-proxy). Можно вернуться к шагу 1. | «Принять результат» |
| 3. Биржа | Выбор/подключение аккаунта биржи + размер позиции. Если нет аккаунта — инлайн-ссылка на Connect exchange. | «Дальше» |
| 4. Sandbox | Запуск в sandbox-стадии. Объяснение: торгуем в «песочнице», копим статистику. | «Запустить sandbox» |
| 5. KPI-gate | Прогресс к критериям промоушена (min trades, winrate, max DD, PnL ≥ threshold). Показ «осталось N сделок». | «Промоутить в live» (активна только при passed) |
| 6. Live | Подтверждение + step-up 2FA (если включён `STEP_UP_REQUIRE_2FA`). Стратегия в live. | «Подтвердить запуск» |

### Статус-трекер стратегии (бэйджи)
- `Draft` (серый) → `Backtested` → `Sandbox` (синий) → `Eligible` (жёлтый, gate passed) → `Live` (зелёный) → `Paused` (красный, авто-пауза KPI-Guard/kill-switch).
- Бэйджи едины для визарда, дашборда Auto-trade и Monitor.

### Технические решения
- Визард — модальный `Dialog` с шаговым индикатором (stepper из примитивов; готового shadcn-stepper нет — собираем из `Badge` + разметки) ИЛИ отдельный маршрут `app/(app)/auto-trade/new/` с под-шагами. **Рекомендация:** отдельный маршрут (визард на полстраницы) — переживает перезагрузку, дольше шаги.
- Состояние визарда — локальный reducer/Zustand-слайс; не плодим глобальный стор без нужды.
- KPI-gate данные — из существующих сервисов `live-auto-trade.ts` (`getStrategyHealth`, `getAutoTradePortfolio`) и promotion-логики (`components/monitor/promotion-utils.ts`).
- Промоушен в live → существующий API + step-up через `lib/api/step-up.ts` (модалка `StepUpModal`).

### Затрагиваемые файлы
- Новый: `app/(app)/auto-trade/new/page.tsx` + `components/auto-trade/launch-wizard/` (по компоненту на шаг).
- Новый: `components/auto-trade/strategy-status-badge.tsx` (единый бэйдж).
- `components/auto-trade/auto-trade-dashboard.tsx` — кнопка «Новая стратегия» → визард; интеграция статус-бэйджа.
- `components/monitor/promotion-gate-panel.tsx` — переиспользовать в шаге 5.
- `lib/api/services/live-auto-trade.ts` — без изменений контракта; при нехватке полей — задокументировать gap, не менять бэкенд.

### Acceptance criteria
- [ ] Из дашборда Auto-trade одной кнопкой запускается визард новой стратегии.
- [ ] Все 6 шагов проходимы; назад/вперёд сохраняет введённые данные.
- [ ] Кнопка «Промоутить в live» неактивна, пока KPI-gate не passed; показывается, чего не хватает.
- [ ] Промоушен в live требует step-up 2FA при `STEP_UP_REQUIRE_2FA=true`.
- [ ] Статус-бэйдж стратегии одинаков в визарде, на дашборде и в Monitor.
- [ ] Прерывание визарда не оставляет «висящую» стратегию в неопределённом состоянии (явный draft/cancel).

## 5. Page Hygiene (все страницы)

Пройти по каждому экрану и применить чек-лист:

| Экран | Файл | Фокус ревизии |
|-------|------|---------------|
| Strategy builder | `app/(app)/strategy/page.tsx`, `components/trading/*` | Свернуть продвинутые параметры под «Advanced»; убрать дубль-подписи у полей формы. |
| Auto-trade | `app/(app)/auto-trade/page.tsx`, `components/auto-trade/*` | Один primary CTA; статус-бэйджи вместо текстовых описаний состояния. |
| Monitor | `app/(app)/monitor/page.tsx`, `components/monitor/*` | KPI-карточки с едиными форматами (`kpi-format.tsx`); явный empty state «нет активных стратегий». |
| Trade | `app/(app)/trade/page.tsx`, `components/trading/cex-dashboard.tsx` | Упростить формы ручной торговли; единые подтверждения. |
| Forecasts | `app/(app)/forecasts/page.tsx` | Каталог-карточки, убрать технические имена файлов из основного текста. |
| Settings/* | `components/settings/*` | Единый layout настроек; короткие подписи; явные success/error тосты. |
| Admin/* | `components/admin-runtime/*`, `components/admin-ai-backtest-config/*` | Доступ только через меню; таблицы с пустыми/loading состояниями. |

### Чек-лист «гигиены» на каждый экран
- [ ] Есть один очевидный primary action.
- [ ] Есть состояния loading / empty / error (не «пустой белый экран»).
- [ ] Нет подписей, дублирующих заголовок/контекст.
- [ ] Технические термины переведены в пользовательские формулировки или скрыты под tooltip.
- [ ] Единые отступы и типографические уровни (через общие токены/utility-классы).
- [ ] Числовые KPI форматируются единообразно (проценты, USDT, ROI — через общие хелперы).

## 8. Code Style (правила реализации)

- **TypeScript strict** — без `any`; типы из `lib/api/openapi-types.ts` как источник правды (руками типы API не пишем).
- **Client vs Server components** — `'use client'` только там, где нужны хуки/интерактив (nav-подсветка, меню, визард). Страницы по умолчанию серверные.
- **Навигация** — только `next/link`; активность — `useSelectedLayoutSegment`, не ручной парсинг URL.
- **Компоненты** — переиспользуем существующие `components/ui/*` (shadcn); новые примитивы ставим через `shadcn add`, не пишем с нуля.
- **Состояние** — локальный стейт по умолчанию; Zustand только для действительно общего состояния (auth, risk-events). Не плодить сторы под визард, если хватает reducer.
- **Стили** — Tailwind utility + существующие токены темы; не вводим инлайн-цвета, используем CSS-переменные темы.
- **Тексты** — короткие, единый тон; технические термины — в tooltip, не в основном потоке.
- **Доступность** — radix/shadcn управляют фокусом; проверяем клавиатурную навигацию меню и визарда.

## 9. Testing Strategy

- **Unit (Vitest + RTL):**
  - `nav-link` — корректная подсветка активного сегмента.
  - `profile-menu` — Admin-пункт скрыт без `hasAdminAccess`; Logout вызывает logout.
  - `strategy-status-badge` — маппинг стадия → визуал/цвет.
  - Шаги визарда — переходы вперёд/назад сохраняют данные; кнопка «Live» заблокирована до gate passed (переиспользовать логику `promotion-gate-panel.test.tsx`).
- **Регрессия контракта:** после любых изменений API-слоя — `npm run gen:api-types` и `npm run build` должны проходить чисто.
- **Ручная проверка (skill `verify`/Chrome MCP):** прогон полного флоу запуска стратегии на дев-стенде; визуальная проверка шапки на desktop/mobile.
- **Качество:** `npm run lint` и `npm run build` — обязательные гейты перед коммитом.

## 10. Boundaries (границы)

### Always (всегда делаем)
- Работаем только в `constructor-front`; бэкенд-контракт не трогаем.
- После правок API-слоя — регенерируем типы и прогоняем build.
- Сохраняем текущую тёмную тему и shadcn-язык (рефреш, не редизайн).
- Сохраняем все существующие функциональные возможности — реорганизуем доступ к ним, не удаляем.
- Соблюдаем step-up 2FA на критичных действиях (запуск live, смена ключей).

### Ask first (спросить перед)
- Удалением любого существующего экрана/маршрута или раздела.
- Введением новой зависимости или новой дизайн-системы.
- Любым изменением, требующим правок бэкенда/OpenAPI.
- Изменением палитры/брендинга (логотип, основные цвета).

### Never (не делаем)
- Не меняем бизнес-логику рисков/исполнения на фронте.
- Не пишем руками типы поверх `openapi-types.ts`.
- Не храним токены/секреты в client-side JS (только httpOnly-куки через route-handlers).
- Не перезаписываем `SPEC.md` (W9) — этот трек живёт в отдельном файле.
- Не вводим полный редизайн с нуля в рамках этого трека.

## 12. План реализации (по фазам)

1. **Фаза 1 — Header** (низкий риск, быстрый эффект): новая шапка, nav-подсветка, dropdown/sheet, перенос статус-индикатора.
2. **Фаза 2 — Status badge + lifecycle**: единый `strategy-status-badge`, интеграция в дашборд/Monitor.
3. **Фаза 3 — Launch wizard**: маршрут `auto-trade/new`, шаги 1–6, KPI-gate, step-up.
4. **Фаза 4 — Page hygiene**: пройтись по чек-листу по всем экранам, empty/loading/error состояния, чистка текстов.
5. **Фаза 5 — Polish**: типографика/отступы, токены, финальный визуальный рефреш, тесты.
